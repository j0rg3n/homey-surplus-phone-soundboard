# SPEC.md — Homey Soundboard

**App name:** Homey Soundboard  
**App icon:** House with sound waves (SVG, house silhouette + emanating arcs, white on colored background)  
**Homey app id:** `com.unofficial.homey-soundboard`  
**Android min SDK:** API 26 (Android 8.0, Galaxy A8 2018 SM-A530)  
**Android target SDK:** API 34  

---

## 1. Overview

Two components:

- **Android app** — manages a library of sound samples, runs a WebSocket server, plays sounds on command, reports playback state back to Homey.
- **Homey app** — represents each paired Android device as a Homey device, exposes Flow cards to trigger/stop sounds and react to playback events.

Communication is a **persistent WebSocket** connection. The Android app is the **server**; Homey is the **client**. Multiple Android devices are supported simultaneously as separate Homey devices.

## 3. Communication Protocol

### 3.1 Transport

- **Protocol:** WebSocket (`ws://`)
- **Default port:** 8765 (user-overridable in Android advanced settings)
- **Android:** Ktor WebSocket server running in a foreground `PlaybackService`
- **Homey:** WebSocket client in `WebSocketClient.js`
- **Heartbeat:** Homey sends `ping` every 15 seconds; Android responds with `pong`. If 3 consecutive pings go unanswered, Homey marks device unavailable and fires `done` with `reason: "connection_lost"` for all in-flight handles.
- **Reconnect:** Homey reconnects with exponential backoff (1s, 2s, 4s, 8s, max 30s).

### 3.2 Message Format

All messages are JSON, newline-delimited. Every message has a `type` field.

#### Downstream (Homey → Android)

```jsonc
// Play a sound
{
  "type": "play",
  "soundId": "uuid-string",
  "volume": 100,           // 0–400%, overrides sample default for this instance
  "handle": "uuid-string"  // generated by Homey, echoed in all callbacks
}

// Stop a sound instance (plays to end of full sample from current position, then stops)
{
  "type": "stop",
  "handle": "uuid-string"
}

// Stop all currently playing sounds (each plays to end of full sample)
{
  "type": "stop_all"
}

// Sent immediately after connect
{
  "type": "hello",
  "version": "1.0"
}

// Heartbeat
{ "type": "ping" }
```

#### Upstream (Android → Homey)

```jsonc
// Response to hello — includes full sound library
{
  "type": "hello_ack",
  "deviceName": "Galaxy A8",
  "version": "1.0",
  "sounds": [
    {
      "id": "uuid-string",
      "name": "Doorbell",
      "durationMs": 2400,
      "loop": false,
      "loopStartMs": 0,
      "loopEndMs": 2400,
      "defaultVolume": 100
    }
  ]
}

// Library changed (sound added, removed, or edited) — full list always sent
{
  "type": "library_update",
  "sounds": [ /* same shape as hello_ack.sounds */ ]
}

// Playback started
{
  "type": "started",
  "handle": "uuid-string",
  "soundId": "uuid-string",
  "soundName": "Doorbell",
  "durationMs": 2400        // 0 for looping sounds (duration is indeterminate)
}

// Playback finished
{
  "type": "done",
  "handle": "uuid-string",
  "soundName": "Doorbell",
  "reason": "completed"     // "completed" | "stopped"
}

// Heartbeat response
{ "type": "pong" }

// Error
{
  "type": "error",
  "handle": "uuid-string",  // null if not playback-related
  "code": "SOUND_NOT_FOUND",
  "message": "Human-readable description"
}
```

### 3.3 Error Codes

| Code | Meaning |
|---|---|
| `SOUND_NOT_FOUND` | `soundId` not in library |
| `HANDLE_NOT_FOUND` | `handle` in stop command not found |
| `PLAYBACK_FAILED` | Audio engine error during play |
| `UNKNOWN_MESSAGE` | Unrecognised message type |

### 3.4 Connection Lifecycle

1. Homey connects via WebSocket.
2. Homey sends `hello`.
3. Android responds with `hello_ack` including full sound library.
4. Homey updates its stored sound list.
5. Normal operation: play/stop commands, started/done callbacks.
6. If Android library changes, Android pushes `library_update`.
7. On disconnect: Homey fires `done` with `reason: "connection_lost"` for all in-flight handles, marks device unavailable, begins reconnect.
8. On reconnect: full `hello` / `hello_ack` handshake. Previous handles are invalid (new session).

## 4. Playback Engine (Android)

### 4.1 Audio API

Use **`AudioTrack`** (API 26+) with **`MediaExtractor`** + **`MediaCodec`** for decoding. This gives sample-accurate loop point control not available with `MediaPlayer` or `SoundPool`.

For sounds with no loop points and under 1MB, `SoundPool` may be used as an optimisation — but `AudioTrack` must be the fallback for all cases.

### 4.2 Supported Formats

MP3, WAV, OGG Vorbis, FLAC.

### 4.3 Concurrency

Multiple sounds play simultaneously using separate `AudioTrack` instances. No protocol-level limit; implementation must handle at least 8 simultaneous instances gracefully.

### 4.4 Playback Handle

- Generated by **Homey** (UUID v4), sent in the `play` message.
- Android uses the handle as the key for the active playback instance.
- Valid only for the current session; handles do not survive app or Homey restart.
- Echoed in all `started`, `done`, and `error` messages.

### 4.5 Volume

- Stored per-sample as `defaultVolume` (0–400%).
- Overridable per-play via `volume` in the `play` message.
- 100% = unity gain.
- Non-linear mapping: `gain = (volume / 100.0) ^ 1.5`, clamped to `[0.0, 1.0]` for `AudioTrack.setVolume()`. This gives a perceptually smooth curve. Values above 100% clip at 1.0 (full device volume) — 400% is a user-facing affordance for future amplification; current implementation silently clamps.
- Note: true amplification above unity gain requires a DSP/mixing layer; this is out of scope for v1. The scale goes to 400% to match the Homey-side setting without requiring a UI change later.

### 4.6 Loop Behavior

- `loop: false` — plays once, fires `done` with `reason: "completed"` at natural end.
- `loop: true` — plays from start, loops between `loopStartMs` and `loopEndMs` indefinitely. Never fires `done` unless stopped.
- On `stop` command: finishes current playthrough from current position to **end of full sample** (not loop end point), then stops. Fires `done` with `reason: "stopped"`.
- `loopStartMs` and `loopEndMs` are validated against sample duration on import. `loopEndMs` must be > `loopStartMs`.

## 5. Android App — UI

### 5.1 Navigation

Two top-level destinations via bottom navigation:

1. **Now Playing** (default on launch)
2. **Library**

Settings accessible via toolbar gear icon from either screen.

### 5.2 Now Playing Screen

- List of currently active playback instances, ordered by start time (newest first).
- Each row:
  - Sound name
  - Progress bar showing elapsed / total duration. For looping sounds: elapsed time with a looping indicator icon instead of a fixed end.
  - Stop button — calls stop on that handle via `PlaybackService`
- Updates in real time via `PlaybackRepository` StateFlow.
- Empty state: "No sounds playing" message.

### 5.3 Library Screen

- Searchable, alphabetically ordered list of all imported sounds.
- Search bar: case-insensitive substring match on name.
- Each row: sound name, duration (formatted as `m:ss`), loop indicator icon if looping.
- Tap row → opens **Sample Edit Sheet**.
- FAB (+) → system file picker filtered to audio MIME types; supports multi-select.

## 6. Android App — Service

### 6.1 PlaybackService

- Foreground service (required on API 26+).
- Persistent notification: "Soundboard active · N sounds playing" (N updates live).
- Starts on app launch, survives backgrounding.
- Hosts both the WebSocket server and the audio engine.
- Only stopped by explicit user action in Settings ("Stop service").
- Notification channel ID: `soundboard_service`.

### 6.2 WebSocket Server

- Built with **Ktor** (`ktor-server-websockets`, `ktor-server-cio`).
- Listens on `0.0.0.0:<port>`.
- Supports multiple simultaneous Homey connections.
- Pushes `library_update` to all connected clients when library changes.
- Routes `started` and `done` only to the client session that sent the originating `play`.

## 8. Homey App

### 8.1 Pairing Flow

1. Custom pairing view: user enters Android device IP and optionally port.
2. Homey attempts WebSocket connection.
3. Sends `hello`, waits for `hello_ack` (timeout: 5 seconds).
4. On success: device added, name from `hello_ack.deviceName`, editable afterwards.
5. On failure: error message shown in pairing view with retry.

### 8.2 Device Settings

| Setting ID | Type | Default | Notes |
|---|---|---|---|
| `ip_address` | string | — | Android device IP |
| `port` | number | `8765` | WebSocket port |
| `reconnect_max_ms` | number | `30000` | Max reconnect backoff in ms |

### 8.3 Capabilities

| Capability | Type | Notes |
|---|---|---|
| `speaker_playing` | boolean | `true` when ≥ 1 sound is playing |
| `volume_set` | number (0–1) | Maps to 0–400% internally as global output multiplier |

### 8.4 Flow Cards

#### Triggers

| ID | Title | Tokens |
|---|---|---|
| `sound_started` | A sound started playing | `sound_name` (string), `handle` (string), `duration_ms` (number) |
| `sound_done` | A sound finished | `sound_name` (string), `handle` (string), `reason` (string) |

#### Conditions

| ID | Title |
|---|---|
| `is_playing` | Soundboard is currently playing |
| `sound_is_playing` | Sound [name selector] is currently playing |

#### Actions

| ID | Title | Args | Side effects |
|---|---|---|---|
| `play_sound` | Play sound | `sound` (dropdown), `volume` (0–400, default 100) | Sets Flow tag `handle` (string) |
| `stop_sound` | Stop sound | `handle` (string tag) | — |
| `stop_all` | Stop all sounds | — | — |
| `set_volume` | Set global volume | `volume` (0–400) | — |

### 8.5 Sound Library Sync

Stored in device store on every `hello_ack` and `library_update`:

```js
{
  sounds: [
    { id: 'uuid', name: 'Doorbell', durationMs: 2400, loop: false }
  ]
}
```

`play_sound` action's `sound` dropdown is populated from this store. When device is offline, last known list is shown with an "(offline)" label.

## 10. `PlaybackHandleStore.js` Interface

In-memory, on the Homey side.

```js
new PlaybackHandleStore()

store.add(handle, { soundId, soundName, startedAt, durationMs })
store.get(handle)           // → entry or null
store.getAll()              // → array
store.remove(handle)
store.clear()               // → array of removed entries (used to fire done-on-disconnect)
store.isPlaying(soundName)  // → boolean
```

## 11. `WebSocketClient.js` Interface

```js
new WebSocketClient({ ip, port, onMessage, onConnect, onDisconnect })

async connect()
async disconnect()
async send(messageObject)   // serializes via MessageProtocol
isConnected()               // → boolean
```

Reconnect with backoff is internal. Fires `onDisconnect` on drop, `onConnect` on (re)establish.

## 13. Testing Strategy

**Homey (Node.js):**
- Unit: `MessageProtocol`, `PlaybackHandleStore` — no mocks needed, pure logic.
- Unit: `WebSocketClient` — mock the underlying `ws` library.
- Integration: `SoundboardDevice` against `MockWebSocketClient` + `MockHomeyDevice` — verify Flow triggers, capability updates, done-on-disconnect for all in-flight handles.
- E2E: Playwright for pairing UI — IP entry, success, timeout error, retry.

**Android (Kotlin):**
- Unit: `PlaybackService` volume mapping (non-linear curve, clamping), loop stop logic, handle lifecycle — mock `AudioTrack`.
- Unit: `MessageProtocol` Kotlin counterpart (JSON serialization/deserialization).
- UI: Espresso for `LibraryFragment` (search filter, edit sheet fields) and `NowPlayingFragment` (progress updates, stop button).

**Coverage targets:**
- Homey overall: > 65%
- `MessageProtocol`, `PlaybackHandleStore`: > 80%
- Android overall: > 65%
